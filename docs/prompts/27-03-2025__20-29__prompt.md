# Промпт: 2FA-челлендж в core-auth, смена пароля/email и UI

## Цель

Реализовать полноценный поток двухфакторной проверки (2FA challenge) в **core-auth**, подключить его к смене пароля и email на **`/profile/settings`**, обеспечить реальное обновление данных в **core-client-info**, **core-auth** и во всех местах, где хранится или используется email пользователя, и улучшить визуальную часть экранов смены пароля и email.

---

## Этап 1 — API в core-auth для 2FA-челленджа

### Что нужно сделать

Добавить HTTP-ручки, которые:

1. **Стартуют** произвольный 2FA-челлендж: создают запись в БД и возвращают клиенту то, что нужно для дальнейших шагов (идентификатор челленджа, сроки жизни и т.д. по договорённости с фронтом).
2. **Завершают** челлендж: принимают введённый пользователем код из **6 цифр**, сверяют с сохранённым челленджем, при успехе помечают челлендж как пройденный и возвращают статус.
3. **Проверяют челлендж для других бэкендов**: отдельная ручка, которую могут вызывать внутренние сервисы, передавая **id челленджа**, чтобы убедиться, что он был успешно пройден (без дублирования логики проверки кода на каждом сервисе).

### Модель челленджа в БД

Для каждой записи челленджа зафиксировать минимум:

| Поле | Описание |
|------|----------|
| **id** | UUID, уникальный идентификатор челленджа |
| **start_at** | Время старта |
| **expiring_at** | Момент, после которого челлендж считается недействительным без успешного прохождения |
| **resolved_at** | Timestamp успешного прохождения (если ещё не пройден — `NULL` или аналог) |
| **code** | Строка из **ровно 6 цифр** (`\d{6}`), сгенерированная при старте |

### Правила попыток и «переотправки»

- На один челлендж даётся **не более 3 попыток** ввода кода.
- **Переотправка** кода/сообщения **не предусмотрена** в рамках этого же челленджа: пользователь не получает новый код без нового старта челленджа (если продуктово не решено иначе — здесь зафиксировано: без переотправки).

### Ожидаемый результат этапа 1

- Согласованный контракт API (пути, методы, тела запросов/ответов, коды ошибок при неверном коде, истечении срока, исчерпании попыток).
- Миграции/схема БД под поля выше и индексы по необходимости (например, по `id`, по `expiring_at` для очистки).

## Этап 2 — Интеграция 2FA в флоу настроек профиля

- На странице **`/profile/settings`** в сценариях **смены пароля** и **смены email** использовать новые ручки 2FA так же, как для любого другого челленджа: старт → ввод кода → успешное завершение перед применением критичного изменения.

---

## Этап 3 — Реальные данные: пароль и email

- Смена пароля и смена email должны идти через **реальные бэкенд-ручки**, без заглушек.
- После успешной операции данные должны быть **согласованы**:
  - в **core-client-info**;
  - в **core-auth**;
  - и **везде**, где фигурирует email пользователя (единый подход: не оставлять рассинхрон между сервисами).

Уточни при реализации источник истины и порядок обновлений (транзакции, события, outbox — по принятой в проекте архитектуре).

## Этап 4 — Вёрстка и UX

### Общее

- Экраны смены пароля и смены email визуально **привести в порядок**: менее «плоско» и шаблонно, в духе существующего дизайн-системы проекта (отступы, типографика, состояния загрузки/ошибок).

### Специальный инпут для 6-значного кода

- **Пустое состояние:** визуально **6 позиций**, оформленных как **точки** (или аналогично «слотам»), без «сырых» шести отдельных полей без стиля.
- **По мере ввода:** на соответствующих позициях точки **заменяются цифрами**.
- **Редактирование с курсором:** поведение как у **одного связного поля** на 6 символов:
  - пользователь может **перемещать курсор** между позициями;
  - пример: введено `123456`, нужно исправить среднюю цифру — можно поставить курсор на третью позицию, удалить/заменить одну цифру **без** обязательного стирания `456` с конца.

Реализация может быть на базе одного `input` с маской/кастомным отображением или доступного компонента — главное сохранить описанное поведение и доступность (фокус, клавиатура, скринридеры — по стандартам проекта).

## Критерии готовности (чеклист)

- [ ] Три ручки в core-auth работают и задокументированы для фронта и других бэкендов.
- [ ] Челлендж хранится с полями `id`, `start_at`, `expiring_at`, `resolved_at`, `code` (`\d{6}`), соблюдены лимит попыток и отсутствие переотправки в рамках одного челленджа.
- [ ] Смена пароля и email на `/profile/settings` проходит через 2FA и реальные API.
- [ ] Данные обновлены в core-client-info, core-auth и остальных релевантных местах для email.
- [ ] Обновлена вёрстка; инпут кода соответствует требованиям к точкам, цифрам и навигации курсором.

## Заметки для исполнителя

- Согласовать с существующими паттернами аутентификации и межсервисных вызовов в репозитории.
- Не дублировать проверку кода на каждом сервисе — использовать ручку проверки по `id` челленджа там, где это уместно.
