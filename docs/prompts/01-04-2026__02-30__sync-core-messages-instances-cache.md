# План: синхронизация экземпляров core-messages через общий Redis

## Контекст

Сервис **core-messages** может работать в нескольких репликах (несколько подов в Kubernetes). События realtime (**WebSocket**) сейчас доставляются через in-memory `ConnectionManager`: активные соединения хранятся только в памяти **текущего** процесса.

**Цель**: при горизонтальном масштабировании гарантировать доставку событий (`new_message`, `message_read`, `message_edited`, `message_deleted`, `new_conversation` и т.д.) всем затронутым пользователям, независимо от того, на каком поде висит их WebSocket.

Общий слой — **Redis** (как уже задумано в инфраструктуре для общего кэша и координации): используем **Pub/Sub** для рассылки событий между экземплярами; локально на каждом поде по-прежнему держим только «свои» WebSocket и отправляем в них после получения сообщения из Redis.

---

## Текущая проблема

| Что происходит                                                             | Почему ломается                                                                                                             |
| -------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| Пользователь A отправляет сообщение через `POST …/messages`                | Запрос обрабатывается на **поде P1**.                                                                                       |
| Бэкенд вызывает `ws_manager.broadcast_to_conversation` / `send_to_user(B)` | Поиск соединений пользователя B идёт в `active_connections` **только на P1**.                                               |
| WebSocket пользователя B подключён к **поду P2**                           | На P1 для `user_id=B` список соединений пустой → **событие не уходит** или уходит не тому инстансу.                         |
| Симптомы                                                                   | У одного из собеседников чат обновляется по WS, у другого — нет; поведение «плавает» при перезапусках подов и балансировке. |

Дополнительно: **sticky sessions** на ingress для `/core-messages/ws` не решают задачу полностью, потому что **разные пользователи** всё равно оказываются на разных подах; а HTTP-запрос создателя сообщения и WS получателя обрабатываются разными репликами.

---

## Целевое решение: общий Redis (Pub/Sub + при необходимости кэш)

Идея: **не полагаться на общую память процессов**, а использовать **один координационный слой**, доступный всем экземплярам core-messages.

### Роль Redis здесь

1. **Pub/Sub (обязательно для целевого решения)**
   - При необходимости отправить событие пользователю `U` под **любым** подом публикуем JSON в канал, например `core-messages:ws:user:{user_id}` (или один канал `core-messages:ws` с полем `target_user_id` в payload — вариант на проектировании).
   - Каждый под core-messages при старте подписывается на нужные паттерны (или на широкий канал и фильтрует локально).
   - Получив сообщение, под **только** вызывает существующую логику отправки в **локальные** `WebSocket` для этого `user_id`. Если у этого поду соединений нет — просто no-op.

2. **Кэш (опционально, тот же Redis)**
   - Уже планируемый общий Redis для профилей/TTL-кэша (см. `29-03-2026__20-23__redis-optimization-client-info.md`) может быть **тем же инстансом**, чтобы не плодить лишнюю инфраструктуру.
   - Для синхронизации WS кэш в классическом смысле (ключ–значение с TTL) **не заменяет** Pub/Sub: события realtime должны доходить **сразу**, без опроса.

### Поток данных (упрощённо)

```text
POST /messages на поде P1
  → бизнес-логика + commit
  → publish(redis, payload для каждого получателя события)
      → все поды (P1, P2, …) получают копию
      → на каждом: если есть локальные WS для user_id → send_text
```

Исключения и оптимизации (на этапе реализации):

- Не дублировать отправку на том же поду дважды: либо всегда только через Redis (проще, единый путь), либо локальная отправка + publish только «для других нод» (сложнее, меньше трафика в Redis).
- Размер payload: передавать уже сериализованный `event` (как сейчас в `send_to_user`), чтобы все поды слали клиентам одинаковый JSON.

## Этапы внедрения

1. **Инфраструктура**
   - Убедиться, что Redis доступен в кластере (и в dev — см. существующий план docker-compose / Helm).
   - Переменные окружения core-messages: `REDIS_URL` (или host/port/password), таймауты, префикс ключей/каналов.

2. **Зависимости**
   - Клиент Redis с поддержкой async Pub/Sub (например `redis.asyncio` для Python 3.11+).

3. **Жизненный цикл приложения**
   - При **startup**: подписка на канал(ы) + фоновая задача чтения сообщений и вызов локального `ConnectionManager.send_to_user` / обёртки.
   - При **shutdown**: корректное закрытие подписки и соединения.

4. **Интеграция с `websocket_manager`**
   - Вынести «доставку наружу» в слой: `emit_to_user(user_id, event)` → `publish` в Redis; локальная доставка остаётся в `send_to_user_local`.
   - Либо: `send_to_user` сначала локально, затем `publish` для других инстансов (если выбрана схема без дубля на своём поде — уточнить в реализации).

5. **Точки вызова**
   - Все текущие вызовы `ws_manager.send_to_user` / `broadcast_to_conversation` должны попадать в новый слой (один раз, без пропусков).

6. **Наблюдаемость**
   - Логи: ошибки подключения к Redis, длина очереди, счётчики «publish без локальных подписчиков» (нормальная ситуация).
   - Метрики (по желанию): latency Redis, количество доставок WS на под.

7. **Тесты**
   - Локально: два процесса core-messages + один Redis, два WS-клиента — проверка доставки между «псевдо-подами».
   - Нагрузочный сценарий не обязателен на первом этапе, но полезен перед продом.

## Риски и ограничения

| Риск                                  | Митигация                                                                                                                                                                                                |
| ------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Redis недоступен                      | Graceful degradation: логировать ошибку; при критичности — health/readiness не проходят, чтобы не принимать трафик без sync. Либо fallback только на локальную доставку с явным предупреждением в логах. |
| Потеря сообщений Pub/Sub при рестарте | Pub/Sub в Redis — fire-and-forget; для чата это обычно приемлемо (истина в БД). Критичные сценарии — не через Pub/Sub без отдельного проектирования.                                                     |
| Дублирование событий                  | Чёткая схема: один путь публикации; при объединении локальной и Redis-доставки — аккуратно исключить двойную отправку на одном клиенте.                                                                  |

---

## Альтернативы (кратко)

- **Одна реплика core-messages** — временная мера, не масштабируется и не отказоустойчива.
- **Только sticky ingress** — не решает доставку между разными пользователями на разных подах.
- **Другой брокер** (NATS, Kafka) — избыточно, если в проекте уже идёт внедрение Redis.

---

## Критерии готовности

- Два и более подов core-messages: оба пользователя в диалоге получают `new_message` по WS независимо от того, на каком поде создано сообщение и на каком висит WS.
- События `message_read` / правки / удаления ведут себя так же кросс-подово.
- Нет регрессии при одной реплике (Redis всё равно используется или путь без Redis отключён флагом — на усмотрение реализации, лучше единый путь через Redis для меньшего расхождения окружений).
