# ADR: Архитектура сервиса расписания (core-schedule)

## Контекст

Сервис управления университетским расписанием. Основная задача — хранение двухнедельного циклического шаблона занятий с поддержкой форс-мажорных изменений и сессионных событий.

## Решение 1: Двухнедельный цикл через `cycle_anchor_date`

### Проблема

Расписание повторяется каждые две недели. Нужен способ определить, какая «неделя цикла» (1 или 2) соответствует произвольной календарной дате.

### Решение

В таблице `semester` хранится `cycle_anchor_date` — понедельник, с которого начинается «Неделя 1». Формула:

```python
delta_days = (target_date - cycle_anchor_date).days
week_number = (delta_days // 7) % 2 + 1  # 1 или 2
```

### Почему не ISO-недели

ISO-недели ломаются на стыке годов и не привязаны к семестру. `cycle_anchor_date` — явная, управляемая привязка.

## Решение 2: Шаблон + переопределения в отдельных таблицах

### Таблицы

- **`schedule_template`** — двухнедельный шаблон (week_number, day_of_week, slot_number)
- **`schedule_override`** — изменения на конкретную дату (cancel / replace)

### Алгоритм формирования расписания на период

1. Загрузить все слоты шаблона для группы/преподавателя
2. Загрузить все override'ы за запрошенный диапазон дат
3. Для каждого дня:
   - Вычислить week_number по формуле выше
   - Взять слоты шаблона для (week_number, day_of_week)
   - Наложить override'ы: `cancel` — убрать слот, `replace` — подставить новые данные
4. Добавить сессионные события (если попадают в диапазон)
5. Отсортировать по slot_number / start_time

### Почему не одна таблица

Шаблон — это «правило», override — «исключение». Разные жизненные циклы: шаблон задаётся раз на семестр, override'ы добавляются по ходу. Разделение упрощает:

- Просмотр чистого шаблона (без override'ов) — один SELECT
- Подсчёт изменений — SELECT COUNT из override'ов
- Bulk-загрузку шаблона при настройке семестра

## Решение 3: Отдельная таблица `session_event`

### Проблема

Зачёты, экзамены и консультации проходят **после окончания семестра** (после `end_date`). Они не являются «изменениями» шаблона — это самостоятельные события.

### Решение

Таблица `session_event` хранит события сессии отдельно от шаблона и override'ов. При формировании расписания на период — события сессии добавляются в результат, если попадают в запрошенный диапазон дат.

### Почему не override

- Сессия выходит за `semester.end_date` — шаблон там не действует, нечего переопределять
- Запрос «покажи все экзамены» — один SELECT по `session_event`, без фильтрации override'ов по lesson_type

## Решение 4: teacher_id (UUID) + локальный кэш `teacher_cache`

### Проблема

Преподаватели «живут» в `core-client-info` (таблица `user`). Расписание должно ссылаться на них по стабильному идентификатору, а не по строке ФИО (которая может измениться).

### Решение

- В таблицах `schedule_template`, `session_event`, `schedule_override` хранится `teacher_id UUID` — это `User.id` из core-client-info.
- Локальная таблица `teacher_cache(id UUID PK, name VARCHAR)` маппит UUID → ФИО для отображения.
- Синхронизация через `POST /admin/teachers/sync` — upsert списка `{id, name}`.
- Все API-ответы возвращают **оба** поля: `teacher_id` + `teacher_name` (из кэша).

### Почему UUID, а не строка

- UUID стабилен при смене ФИО (опечатка, замужество)
- Позволяет однозначно идентифицировать преподавателя
- Teacher-эндпоинты принимают UUID в path: `/teachers/{teacher_id}/schedule`

### Остальные строковые связи

Группы (`group_name`) и предметы (`subject_name`) по-прежнему хранятся как строки — в core-client-info для них нет нормализованных таблиц с UUID.

## Решение 5: Авторизация — role check в handler

### Проблема

`AuthMiddleware` из `authorization-py` поддерживает только `auth_level: none | valid`. Нужна проверка роли (admin/moderator) для CRUD-эндпоинтов.

### Решение

FastAPI dependency `require_admin` проверяет `user["role"]` из JWT:

```python
async def _require_admin(user = Depends(get_current_user)):
    if user.get("role") not in ("admin", "moderator"):
        raise HTTPException(403, "Admin access required")
    return user

AdminUser = Annotated[dict, Depends(_require_admin)]
```

Read-эндпоинты (`/groups/...`, `/teachers/...`) доступны любому авторизованному пользователю. Admin-эндпоинты (`/admin/...`) требуют роль admin или moderator.
