# Лабораторная работа №4

## 1. Цель работы

Целью данной лабораторной работы является применение методов тест-дизайна, а именно анализа эквивалентных классов и тестирования граничных значений, при разработке набора тест-кейсов для формы заказа справок в пользовательском интерфейсе системы diploma.sewaca.ru.

В рамках работы необходимо проанализировать поведение полей формы «Тип справки», «Где получить справку», чекбокса «Только виртуальная справка (PDF)», а также кнопки отправки формы «Заказать». Требуется выделить валидные и невалидные классы входных данных, определить граничные случаи и рассчитать оптимальное количество тестов, необходимое для покрытия ключевой функциональности формы.

## 3. Описание эквивалентных классов

### Параметр 1: Тип справки

EC1.1 (Позитивный класс): выбрано существующее значение из списка доступных типов справок.  
Примеры: `РЖД`, `По месту работы`, `По месту работы родителей`, `Для военкомата`, `О стипендии`.

EC1.2 (Позитивный класс): введено пользовательское значение, отсутствующее в списке, но сохранённое как отдельная опция.  
Пример: `Справка для архива`.

EC1.3 (Позитивный класс): введено значение с пробелами по краям, которые удаляются при обработке.  
Пример: `   По месту работы   `.

EC1.4 (Негативный класс — пустое значение): поле не заполнено.  
Пример: ``.

EC1.5 (Негативный класс — только пробелы): после удаления пробелов значение становится пустым.  
Пример: `   `.

EC1.6 (Негативный класс — промежуточный ввод): пользователь только начал ввод, но итоговое значение ещё не подтверждено логикой формы.  
Пример: `По`.

Граничные значения для параметра «Тип справки»:

- пустая строка → ``
- строка из пробелов → `   `
- минимально осмысленный ввод → `РЖД`
- ввод, совпадающий с началом нескольких вариантов → `По`
- ввод, отсутствующий в справочнике → `zzz`

### Параметр 2: Где получить справку

EC2.1 (Позитивный класс): выбран корректный пункт выдачи из списка, соответствующего выбранному типу справки.  
Пример: `Студенческий отдел кадров СПбГУТ — пр. Большевиков, д.22, каб. 602/1`.

EC2.2 (Позитивный класс): выбран другой допустимый пункт выдачи из того же списка.  
Пример: `Студенческий отдел кадров СПбКТ — наб. реки Мойки, 61, каб. 260`.

EC2.3 (Позитивный класс): после смены типа справки поле автоматически устанавливается в первое доступное значение нового списка.

EC2.4 (Негативный класс — тип не выбран): поле недоступно, так как ещё не определён список пунктов выдачи.

EC2.5 (Негативный класс — значение не выбрано при обязательности поля): чекбокс виртуальной справки выключен, но пункт выдачи отсутствует.

EC2.6 (Позитивный класс): при включённой опции «Только виртуальная справка (PDF)» поле скрыто и не участвует в валидации.

Граничные состояния для параметра «Где получить справку»:

- поле заблокировано до выбора типа;
- после выбора типа автоматически подставляется первое значение;
- при смене типа ранее выбранный пункт заменяется новым первым пунктом;
- при включении режима PDF поле полностью скрывается.

### Параметр 3: Только виртуальная справка (PDF)

EC3.1 (Позитивный класс): значение `true`, форма оформляет заказ только в электронном виде.

EC3.2 (Позитивный класс): значение `false`, форма требует выбор пункта выдачи бумажной справки.

EC3.3 (Позитивный класс): переключение из `false` в `true` скрывает поле выбора места получения.

EC3.4 (Позитивный класс): переключение из `true` в `false` снова делает поле выбора места получения обязательным.

### Параметр 4: Кнопка «Заказать»

EC4.1 (Позитивный класс): кнопка активна, если заполнен тип справки и выполнены все обязательные условия формы.

EC4.2 (Негативный класс): кнопка неактивна, если тип справки отсутствует.

EC4.3 (Негативный класс): кнопка неактивна, если чекбокс PDF выключен и не выбран пункт выдачи.

EC4.4 (Позитивный класс): кнопка активна, если тип выбран и чекбокс PDF включён, даже без выбора пункта выдачи.

Граничные состояния для кнопки:

- disabled при пустой форме;
- disabled при частично заполненной форме;
- active при валидной бумажной справке;
- active при валидной виртуальной справке.

### Параметр 5: Фильтрация списка типов

EC5.1 (Позитивный класс): пустой ввод показывает полный список типов с сервера.

EC5.2 (Позитивный класс): ввод фильтрует значения по началу строки без учёта регистра.  
Пример: `По`, `по`.

EC5.3 (Позитивный класс): если совпадений нет, пользовательский ввод появляется как отдельная опция.  
Пример: `zzz`.

EC5.4 (Негативный класс): ввод не должен фильтровать по середине строки, если логика поддерживает только совпадение по началу названия.

## 4. Расчет количества тестов

Форма заказа справки содержит 5 основных проверяемых блоков:

- тип справки;
- пункт получения справки;
- чекбокс «Только виртуальная справка (PDF)»;
- кнопка «Заказать»;
- фильтрация списка значений.

### 1. Классический подход (без оптимизации)

Если учитывать все основные комбинации состояний:

- Тип справки: 6 классов
- Где получить справку: 6 классов
- PDF-чекбокс: 4 состояния/перехода
- Кнопка отправки: 4 класса
- Фильтрация: 4 класса

Итого:

`6 × 6 × 4 × 4 × 4 = 2304 тестов`

Полный перебор комбинаций даёт слишком большое количество проверок и является неэффективным.

### 2. По фактической таблице

В таблице тест-кейсов рассмотрено:

- Тип справки: 8 тестов
- Где получить справку: 6 тестов
- PDF-чекбокс: 4 теста
- Кнопка отправки: 6 тестов
- Фильтрация: 4 теста

Итого:

`8 + 6 + 4 + 6 + 4 = 28 тестов`

### 3. Оптимизированный подход (рекомендуемый)

Используем по одному представителю из каждого эквивалентного класса, а также основные граничные значения и переходы состояний.

Параметр «Тип справки»:

- 2 позитивных теста (значение из списка, пользовательское значение)
- 3 негативных/граничных теста (пусто, пробелы, промежуточный ввод)

Итого: 5 тестов

Параметр «Где получить справку»:

- 2 позитивных теста (два валидных пункта)
- 2 теста состояния (поле заблокировано, поле скрыто в PDF-режиме)
- 1 тест автоподстановки

Итого: 5 тестов

Параметр «PDF»:

- 2 основных состояния (`true`, `false`)
- 2 перехода между состояниями

Итого: 4 теста

Параметр «Кнопка Заказать»:

- 2 позитивных теста
- 2 негативных теста

Итого: 4 теста

Параметр «Фильтрация»:

- 1 тест пустого ввода
- 1 тест фильтрации по началу строки
- 1 тест без учёта регистра
- 1 тест пользовательского варианта

Итого: 4 теста

Общее количество тестов:

`5 + 5 + 4 + 4 + 4 = 22 теста`

Эффективность оптимизации:

`(2304 − 22) / 2304 × 100% ≈ 99.05%`

Таким образом, применение методов тест-дизайна позволяет сократить количество тестов более чем на 99% без потери покрытия ключевых сценариев формы.

## 5. Выводы по работе

В ходе выполнения лабораторной работы были применены методы тест-дизайна: анализ эквивалентных классов и тестирование граничных значений.

Были определены эквивалентные классы для следующих элементов формы:

- тип справки;
- место получения справки;
- режим виртуальной справки;
- кнопка отправки формы;
- механизм фильтрации списка значений.

На основе анализа была сформирована таблица из 28 тест-кейсов, охватывающая:

- корректные пользовательские действия;
- некорректные сценарии ввода;
- переходы между состояниями интерфейса;
- граничные случаи поведения формы.

Полный перебор всех комбинаций потребовал бы 2304 теста, что является неэффективным с практической точки зрения.

Применение оптимизированного подхода позволило сократить количество тестов до 22, при этом сохранив покрытие основных эквивалентных классов, переходов состояний и критически важных пользовательских сценариев.

Таким образом, использование методов тест-дизайна позволяет существенно уменьшить объём тестирования, сохранить качество проверки интерфейса и повысить эффективность процесса разработки и контроля качества программного продукта.
